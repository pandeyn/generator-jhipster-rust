# Configuration Reference

This document describes all configuration options available in applications generated by the JHipster Rust blueprint. Configuration is managed through environment variables in the `.env` file, with optional layered configuration via Consul KV when external configuration is enabled.

## Configuration Layers

The configuration system supports multiple layers. The exact layers depend on whether external configuration is enabled.

### Without External Configuration

```
Environment variables  (highest priority)
        ↓
   .env file           (loaded at startup via dotenvy)
        ↓
Compiled defaults      (hardcoded fallbacks)
```

### With External Configuration (Consul KV)

```
Environment variables                     (highest priority)
        ↓
   .env file
        ↓
Consul KV profile-specific               (e.g., config/<app>/application-dev)
        ↓
Consul KV base                           (e.g., config/<app>/application)
        ↓
Compiled defaults                        (lowest priority)
```

When a value is set in both Consul KV and the `.env` file, the `.env` / environment variable always wins.

## Application Settings

Core application settings that control the server behavior.

| Variable             | Description                                                    | Default                                            |
| -------------------- | -------------------------------------------------------------- | -------------------------------------------------- |
| `APP_NAME`           | Application name (used in logs and metadata)                   | `<baseName>`                                       |
| `APP_ENV`            | Environment name (`development`, `staging`, `production`)      | `development`                                      |
| `APP_PORT`           | HTTP server port                                               | `8080` (monolith/gateway) or `8081` (microservice) |
| `APP_HOST`           | Bind address for the HTTP server                               | `0.0.0.0`                                          |
| `APP_HTTPS`          | Set to `true` when running behind HTTPS (gateway/monolith)     | `false`                                            |
| `SERVE_STATIC_FILES` | Enable serving the SPA UI from Rust backend (gateway/monolith) | `false`                                            |
| `STATIC_FILES_DIR`   | Directory containing the built frontend app                    | `./static`                                         |
| `RUST_LOG`           | Logging level filter (uses `tracing` format)                   | `info,<baseName>=debug,tower_http=debug`           |

## External Configuration (Consul KV)

Optional feature available when Consul service discovery is enabled. When enabled, the application loads configuration from Consul's KV store at startup and optionally watches for changes at runtime.

### Environment Variables

| Variable                       | Description                                             | Default             |
| ------------------------------ | ------------------------------------------------------- | ------------------- |
| `CONSUL_ENABLE_CONFIG`         | Enable loading configuration from Consul KV             | `true`              |
| `CONSUL_CONFIG_KEY_PREFIX`     | KV prefix for configuration keys                        | `config/<baseName>` |
| `APP_PROFILE`                  | Active configuration profile (`dev`, `staging`, `prod`) | `dev`               |
| `CONSUL_CONFIG_WATCH_ENABLED`  | Enable hot-reload watching of KV changes                | `true`              |
| `CONSUL_CONFIG_WATCH_INTERVAL` | Watch polling interval in seconds                       | `55`                |

### How It Works

1. **Startup**: The application loads configuration from two Consul KV paths:

   - `config/<baseName>/application` — base configuration
   - `config/<baseName>/application-{APP_PROFILE}` — profile-specific overrides

2. **Layering**: Values from the profile-specific path override the base path. Environment variables and `.env` values override both.

3. **Hot-Reload**: When `CONSUL_CONFIG_WATCH_ENABLED=true`, a background watcher polls Consul KV for changes and updates the `RemoteConfig` in memory without restarting the application.

### Hot-Reloadable Settings (RemoteConfig)

These settings can be updated at runtime via Consul KV without restarting:

| Consul KV Key                            | Description                                        | Default |
| ---------------------------------------- | -------------------------------------------------- | ------- |
| `logging.level`                          | Log level (`debug`, `info`, `warn`, `error`)       | `info`  |
| `database.pool_size`                     | Database connection pool size                      | `10`    |
| `health.interval_seconds`                | Health check interval                              | `10`    |
| `circuit_breaker.failure_rate_threshold` | Failure rate to open circuit (0.0-1.0)             | `0.5`   |
| `circuit_breaker.sliding_window_size`    | Number of calls in sliding window                  | `100`   |
| `circuit_breaker.wait_duration_secs`     | Seconds circuit stays open                         | `60`    |
| `circuit_breaker.request_timeout_ms`     | Request timeout in milliseconds                    | `30000` |
| `feature.*`                              | Custom feature flags (e.g., `feature.new_ui=true`) | None    |

### Profile Configuration Files

When deployed with Docker Compose or Kubernetes, configuration profiles are loaded from YAML files into Consul KV:

- `docker/central-server-config/application.yml` — Base configuration
- `docker/central-server-config/application-dev.yml` — Development overrides
- `docker/central-server-config/application-prod.yml` — Production overrides

### Managing Configuration

You can read and update configuration directly via the Consul API:

```bash
# Read a config value
curl http://localhost:8500/v1/kv/config/<baseName>/application/logging/level?raw

# Set a config value
curl -X PUT -d 'debug' http://localhost:8500/v1/kv/config/<baseName>/application/logging/level

# Set a feature flag
curl -X PUT -d 'true' http://localhost:8500/v1/kv/config/<baseName>/application/feature.dark_mode
```

## Vault Secrets Management

Available when Vault is selected as the secrets management solution. Requires external configuration to be enabled. Secrets are loaded at startup and set as environment variables.

| Variable            | Description                                   | Default                  |
| ------------------- | --------------------------------------------- | ------------------------ |
| `VAULT_ENABLED`     | Enable Vault integration                      | `true`                   |
| `VAULT_ADDR`        | Vault server address                          | `http://localhost:8200`  |
| `VAULT_AUTH_METHOD` | Authentication method (`token` or `approle`)  | `token`                  |
| `VAULT_TOKEN`       | Vault token (for `token` auth method)         | `myroot`                 |
| `VAULT_ROLE_ID`     | AppRole role ID (for `approle` auth method)   | None                     |
| `VAULT_SECRET_ID`   | AppRole secret ID (for `approle` auth method) | None                     |
| `VAULT_SECRET_PATH` | Path to read secrets from                     | `secret/data/<baseName>` |
| `VAULT_AUTO_RENEW`  | Enable automatic token renewal                | `true`                   |

### Authentication Methods

- **Token** (`VAULT_AUTH_METHOD=token`): Use `VAULT_TOKEN` directly. Suitable for development.
- **AppRole** (`VAULT_AUTH_METHOD=approle`): Use `VAULT_ROLE_ID` and `VAULT_SECRET_ID`. Recommended for production.

### How Secrets Work

1. On startup, the application reads all secrets from `VAULT_SECRET_PATH`
2. Each secret key-value pair is set as an environment variable (if not already set)
3. Environment variables always take precedence over Vault values
4. When `VAULT_AUTO_RENEW=true`, the Vault token is automatically renewed before expiration

## Production Checklist

Before deploying to production, review these critical settings:

- [ ] **`JWT_SECRET`**: Replace with a strong, random secret (minimum 256 bits)
- [ ] **`APP_ENV`**: Set to `production`
- [ ] **`APP_PROFILE`**: Set to `prod` (when using external config)
- [ ] **`DATABASE_URL`**: Update with production database credentials
- [ ] **`VAULT_AUTH_METHOD`**: Switch to `approle` with unique role/secret IDs (when using Vault)
- [ ] **`CONSUL_ACL_TOKEN`**: Set if Consul ACLs are enabled
- [ ] **`KAFKA_SECURITY_PROTOCOL`**: Enable `ssl` or `sasl_ssl` for production Kafka
- [ ] **`RUST_LOG`**: Set to `info` or `warn` (avoid `debug` in production)
